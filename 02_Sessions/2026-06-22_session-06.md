## Gebaut: Bot-/Abuse-Schutz (Schicht 1+2)

**Neu `src/lib/rate-limit.ts`** — geteilte, getestete Util:
- `getClientIp(headers)` — `cf-connecting-ip` → `x-forwarded-for` → `x-real-ip` → `'unknown'`
- `createRateLimiter(limit, windowMs)` — Fixed-Window per-IP, lazy Eviction-Sweep (kein Memory-Leak)
- Ersetzt das duplizierte Rate-Limit in `/api/validate` (DRY).

**`/api/subscribe` gehärtet:**
- Rate-Limit **5/IP/Stunde** → 429 (löst Session-05-Finding #1: E-Mail-Bombing-Vektor)
- **Honeypot** (`website`-Feld): bei Befüllung silent 200 ohne Brevo-Call (kein Detektionssignal an Bot)
- **Time-Trap**: Submit < 2s → silent 200 (Client-Timestamp, bewusst nur „cheap filter")

**`/api/validate`** auf geteilte Util umgestellt (10/IP/Min, Verhalten unverändert).

**`EmailCaptureSection.tsx`**: verstecktes Honeypot-Feld (off-screen + `aria-hidden` + `tabIndex -1`) + Render-Timestamp `t0` im Submit.

**Erste Tests im Repo** `src/lib/rate-limit.test.ts` (6×): Limit-Durchsetzung, IP-Trennung, Fenster-Reset, `getClientIp`-Prioritäten.
- Dabei: vitest-Environment war auf `jsdom` konfiguriert, Paket aber **nicht installiert** → auf `node` gestellt (aktuelle Tests sind reine Logik).
