## Gefundene & behobene Defekte

### 🔴 SECURITY — SSRF: Redirect-Bypass + DNS-Rebinding
**Befund:** `route.ts` nutzte `redirect: 'follow'` (folgt bis zu 20 Redirects automatisch), die Konstante `MAX_REDIRECTS = 2` war deklariert, aber **nie verwendet**. Redirect-Ziele wurden nicht gegen die SSRF-Blocklist geprüft. Zusätzlich prüfte `isBlockedUrl` nur den URL-String per Regex — eine öffentliche Domain, die per DNS auf `127.0.0.1` oder `169.254.169.254` (Hetzner/Cloud-Metadata) zeigt, wäre durchgerutscht.

**Risiko:** Angreifer hätte den Server interne Adressen (inkl. Cloud-Metadata-Endpoint) abrufen lassen können → Datenexfiltration. Gleichzeitig **Spec-Verletzung** (Pflichtenheft S7.1: „Redirects: max. 2").

**Fix:**
- Neue `safeFetch()` in `route.ts`: manuelles Redirect-Following, max. 2 Hops, **jeder Hop** vorab gegen Blocklist UND aufgelöste IPs geprüft.
- `ssrf.ts` erweitert: `isPrivateIp()` (IPv4 + IPv6, inkl. CGNAT, Multicast, IPv4-mapped IPv6) + `resolvesToPrivate()` (DNS-Lookup `all: true`, fail-closed bei Auflösungsfehler).
- `export const runtime = 'nodejs'` erzwungen — DNS-Auflösung ist auf der Edge-Runtime nicht verfügbar.
- Neuer Status `REDIRECT_LIMIT` wird sauber an den Client gemappt.

### 🟡 UX-BREAK — Content-Type zu streng
**Befund:** Validator wies jede Antwort mit Content-Type ≠ `application/json` mit `FORMAT_ERROR` ab. Aber unsere **eigene** Zero-Tech-Anleitung empfiehlt GitHub Gist / CDNs, die `.json` als `text/plain` oder `application/octet-stream` ausliefern. Legitime Nutzer-Dateien wären fälschlich abgelehnt worden.

**Fix:** Content-Type ist jetzt **beratend**; maßgeblich ist „parst als JSON + Schema-Match". Genuine Nicht-JSON-Bodies scheitern weiterhin sauber mit `FORMAT_ERROR`. Bewusste, dokumentierte Abweichung von Pflichtenheft S7.1 Step 5 (im Code kommentiert, im DASHBOARD als zu ratifizierende Spec-Änderung geflaggt).

### 🟡 UX-BUG — Generator: human_review Radio-Doppelmarkierung
**Befund:** „Ja, immer" und „Ja, teilweise" hatten beide `bool: true`. Klick auf „teilweise" markierte optisch „immer" (beide leuchteten auf), weil die Highlight-/checked-Logik nur den Bool verglich.

**Fix:** Neues Feld `humanReviewChoice: string` in `FormData` trackt die tatsächlich gewählte Option. `checked` und Highlight vergleichen jetzt den Optionswert. Reset in „Neue JSON erstellen" angeglichen.

### 🟢 HARDENING — Security-Header unvollständig
**Befund:** `next.config.js` hatte nur 3 Header; CSP, HSTS, Permissions-Policy fehlten.
**Fix:** Vollständiger Header-Satz: Content-Security-Policy (self + unsafe-inline für Next-Hydration), Strict-Transport-Security (2 J, preload), Permissions-Policy, X-Frame-Options DENY, X-Content-Type-Options, Referrer-Policy, X-DNS-Prefetch-Control off. `poweredByHeader: false`.

### 🟢 ROBUSTHEIT — Rate-Limit Memory-Leak
**Befund:** `ipCounts`-Map evingierte abgelaufene Einträge nie → unbegrenztes Wachstum über die Prozesslaufzeit.
**Fix:** `sweepExpired()` läuft max. 1×/Minute und entfernt abgelaufene Einträge.

### 🟢 SEO — sitemap.xml fehlte
**Befund:** `robots.txt` referenzierte `https://aiclaration.de/sitemap.xml`, die Datei existierte aber nicht → 404.
**Fix:** `public/sitemap.xml` erstellt (/, /check, /generate, /validate, /spec; Legal-Seiten ausgeschlossen).

### Build-Fehler behoben
`while (true)` in der Body-Lese-Schleife verletzte ESLint `no-constant-condition` und ließ den Build fehlschlagen → auf `while (streaming)` umgestellt.

## Offen / kein Code-Defekt, aber wichtig

- **DNS-Rebinding-Restrisiko (TOCTOU):** Zwischen IP-Check und Fetch existiert theoretisch ein Zeitfenster. Für einen öffentlichen Well-Known-Validator akzeptabel; echte Eliminierung erfordert Fetch-by-IP mit Host-Header (bricht TLS-SNI). Bewusst nicht für V1.
- **CSP `unsafe-inline`:** Next.js-Hydration braucht es ohne Nonce-Middleware. Später mit Nonces härten.
- **Rate-Limit ist per-Prozess:** Bei Multi-Instanz-Deployment (mehrere Coolify-Replicas) nicht global. Für V1 Single-Instance ausreichend.
- **Rechtliche `[PLACEHOLDER]`** weiterhin Founder-Aufgabe (siehe DASHBOARD).
